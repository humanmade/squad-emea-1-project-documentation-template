# Architecture decision records

**A log of significant technical decisions made on this project.** Each ADR captures what was decided, why, and what alternatives were considered.

## What belongs here

Record decisions that are non-obvious, involve trade-offs, or would take meaningful time to re-derive if questioned later. Not every technical choice warrants an ADR — focus on decisions that shape the architecture or constrain future work.

Good candidates:

- Choice of plugin, service, or third-party integration
- Decisions to deviate from the HM default approach (e.g. using a custom block where a pattern would normally suffice)
- Caching strategy, auth model, or data storage approach
- Anything where "why did we do it this way?" is a question someone will reasonably ask

## Status values

- **Proposed** — under discussion, not yet adopted
- **Accepted** — the decision is in effect
- **Superseded** — replaced by a later ADR (link to it)
- **Deprecated** — no longer applies
